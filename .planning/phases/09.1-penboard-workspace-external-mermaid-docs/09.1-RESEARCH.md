# Phase 09.1: PenBoard Workspace & External Mermaid Docs - Research

**Researched:** 2026-03-21
**Domain:** Workspace directory structure, Mermaid rendering in React, MCP tool expansion, editor layout tabs
**Confidence:** HIGH

## Summary

Phase 09.1 introduces a `.penboard/` workspace directory alongside `.pb` files, adds a Canvas/Flow top-level tab system to the editor, renders mermaid diagrams from `.penboard/flows/*.md` files in a scrollable view-only Flow tab, and adds MCP tools for AI agents to read/write flows and context docs.

The project already has mermaid.js v11.13.0 installed and actively used in `WorkflowPanel` with the `mermaid.render()` API -- the exact same API needed for the Flow tab. The existing MCP tool architecture (`tool-registry.ts` + individual tool files) provides a clear pattern for adding the 5 new workspace tools. The PenDocument type needs a new `workspace` field, and the editor layout needs a top-level tab system above the toolbar.

**Primary recommendation:** Reuse the existing mermaid.js rendering approach from `WorkflowPanel`, parse `.md` files client-side to extract mermaid blocks and description text, render each flow as a section with TOC navigation. MCP tools follow the established pattern in `src/mcp/tools/`.

### Pattern 1: Top-Level Tab System (Canvas / Flow)

**What:** A tab bar component rendered between `TopBar` and the main editor content area in `editor-layout.tsx`. When "Canvas" tab is active, the current editor (SkiaCanvas + toolbar + panels) renders. When "Flow" tab is active, a scrollable mermaid viewer renders instead.

**When to use:** The tabs must be top-level, above the toolbar, as decided in CONTEXT.md.

**Implementation approach:**

### Pattern 2: Mermaid Rendering (Reuse from WorkflowPanel)

**What:** Use `mermaid.render(id, definition)` to convert mermaid text into SVG, then inject via `dangerouslySetInnerHTML` or `containerRef.innerHTML`. This is the exact pattern used in `workflow-panel.tsx`.

**Key detail:** mermaid.js requires unique IDs per render call. Use a counter or nanoid to ensure uniqueness.

```typescript
// Existing pattern from workflow-panel.tsx line 34-36:

const { svg } = await mermaid.render(id, mermaidText)
containerRef.current.innerHTML = svg
```

**For Flow view, adapt to render multiple diagrams:**
```typescript
// Each flow section gets its own unique mermaid render ID
const renderMermaid = async (text: string, containerId: string) => {
  const { svg } = await mermaid.render(containerId, text)
  return svg
}
```

### Pattern 3: Markdown Parsing for Flow Files

**What:** Each `.penboard/flows/*.md` file contains markdown with embedded mermaid code blocks. Parse the markdown to extract: title (H1), description paragraphs, and mermaid code blocks.

**Simple approach -- no new dependencies needed:**
```typescript
interface ParsedFlow {
  title: string        // From H1 heading
  description: string  // All non-mermaid content
  mermaidBlocks: string[] // Extracted ```mermaid blocks
  rawContent: string   // Full file content
}

function parseFlowFile(content: string): ParsedFlow {
  const lines = content.split('\n')
  const title = lines.find(l => l.startsWith('# '))?.replace(/^#\s+/, '') ?? 'Untitled'

  const mermaidBlocks: string[] = []
  let inMermaid = false
  let currentBlock = ''

  for (const line of lines) {
    if (line.trim() === '```mermaid') {
      inMermaid = true
      currentBlock = ''
    } else if (inMermaid && line.trim() === '```') {
      inMermaid = false
      mermaidBlocks.push(currentBlock.trim())
    } else if (inMermaid) {
      currentBlock += line + '\n'
    }
  }

  // Description = everything except H1 and mermaid blocks
  const description = content
    .replace(/^#\s+.*/m, '')
    .replace(/```mermaid[\s\S]*?```/g, '')
    .trim()

  return { title, description, mermaidBlocks, rawContent: content }
}
```

### Pattern 4: Workspace Path Resolution

**What:** The `.pb` file stores a `workspace` field as a relative path to the project root containing `.penboard/`. The MCP tools resolve this path to read/write workspace files.

**Key considerations:**
- In **MCP (Node.js) context**: Use `path.resolve(path.dirname(filePath), doc.workspace)` to resolve the workspace root from the `.pb` file's location.
- In **browser/Electron context**: Need a server API endpoint to read workspace files, since the browser cannot directly access the filesystem.
- For **live canvas mode**: The document-store tracks `filePath` (Electron) which can resolve workspace location. Browser mode needs server-side endpoint.

```typescript
// In MCP tools:
import { dirname, resolve, join } from 'node:path'

function resolveWorkspacePath(pbFilePath: string, workspace: string): string {
  return resolve(dirname(pbFilePath), workspace)
}

function resolveFlowPath(workspaceRoot: string, flowName: string): string {
  return join(workspaceRoot, '.penboard', 'flows', `${flowName}.md`)
}
```

### Pattern 5: Server API for Flow Data (Browser Access)

**What:** The browser cannot read `.penboard/flows/` directly. Add Nitro server API endpoints that the Flow tab calls to load flow data. These endpoints use the MCP sync URL pattern already established.

```
GET /api/mcp/flows              -> list all flows
GET /api/mcp/flows/:name        -> read a flow file
POST /api/mcp/flows/:name       -> write a flow file (for MCP live sync)
```

**Alternative (simpler):** Since the Flow tab is view-only, the MCP tools write files, and the UI just needs to display them. A single endpoint that returns all flows as JSON is sufficient:

```
GET /api/workspace/flows -> { flows: [{ name, title, content }] }
```

### Anti-Patterns to Avoid
- **Don't use `rehype-mermaid` plugin:** It requires SSR or build-time rendering. The project uses client-side mermaid.render() which is simpler and already proven in WorkflowPanel.
- **Don't embed mermaid content in PenDocument:** The whole point of this phase is to externalize flows out of `.pb` files. Keep them in `.penboard/flows/`.
- **Don't create a writable mermaid editor:** CONTEXT.md explicitly says view-only in UI, AI edits via MCP.
- **Don't parse markdown with complex AST libraries:** Simple regex parsing of H1, description, and mermaid blocks is sufficient for the well-defined `.md` format.

## Common Pitfalls

### Pitfall 1: `.penboard/` Directory Name Collision
**What goes wrong:** The constant `PORT_FILE_DIR_NAME` in `src/constants/app.ts` is already `'.penboard'`, used for the port file at `~/.penboard/.port` (in user HOME).
**Why it happens:** Same directory name, different locations.
**How to avoid:** The workspace `.penboard/` is at the project root (next to `.pb` file), while the port file `.penboard/` is in `$HOME`. These are different paths. No collision, but document this clearly. The MCP tools resolve workspace path relative to the `.pb` file, not the home directory.
**Warning signs:** If someone confuses the two, workspace operations could touch the port file directory.

### Pitfall 2: Mermaid Render ID Uniqueness
**What goes wrong:** Multiple mermaid diagrams on the same page use duplicate render IDs, causing rendering failures or overwritten SVG.
**Why it happens:** `mermaid.render(id, text)` requires globally unique IDs per render call. If two Flow sections use the same ID, one overwrites the other.
**How to avoid:** Use an incrementing counter pattern (like `workflow-panel.tsx`) combined with the flow name: `flow-${flowName}-${counter}`.
**Warning signs:** Some mermaid diagrams show blank or show the wrong diagram.

### Pitfall 3: Mermaid Security Level
**What goes wrong:** Click handlers or embedded HTML in mermaid diagrams may not work.
**Why it happens:** Mermaid defaults to `securityLevel: 'strict'` which disables click events.
**How to avoid:** The existing WorkflowPanel uses `securityLevel: 'loose'`. For the Flow tab (view-only), `'strict'` is appropriate since no click interaction is needed. Use a separate mermaid.initialize() call or pass config per render if different security levels are needed.
**Warning signs:** Interactive features in mermaid diagrams don't work.

### Pitfall 4: Server-Side Flow File Access from Browser
**What goes wrong:** The Flow tab component tries to use `fs.readFile` in the browser, causing a runtime error.
**Why it happens:** Browser code cannot access the filesystem directly.
**How to avoid:** Create a Nitro server API endpoint (e.g., `GET /api/workspace/flows`) that reads `.penboard/flows/` and returns the data as JSON. The Flow tab fetches this endpoint. In Electron, use IPC or the same HTTP endpoint (Nitro runs as a child process).
**Warning signs:** Module resolution errors mentioning 'fs' or 'node:fs' in browser bundle.

### Pitfall 5: workspace Field Missing on Existing Documents
**What goes wrong:** Opening an existing `.pb` file without `workspace` field causes the Flow tab to fail or show empty.
**Why it happens:** Legacy files don't have the `workspace` field.
**How to avoid:** Treat `workspace` as optional. If missing, the Flow tab shows an empty state with instructions ("No workspace linked"). MCP tools can auto-create `.penboard/` and set the workspace field on first `write_flow` call.
**Warning signs:** Flow tab crashes or shows error for old documents.

### Pitfall 6: MCP Live Canvas vs File Mode for Workspace
**What goes wrong:** Workspace tools fail in live canvas mode because there's no file path to resolve the workspace from.
**Why it happens:** Live canvas mode (`live://canvas`) has no disk path, so relative workspace resolution fails.
**How to avoid:** For live canvas mode, resolve workspace from the `filePath` stored in the document store (via the sync API). Add a new sync endpoint `GET /api/mcp/workspace-path` that returns the current document's file path if saved. If the document is unsaved (no filePath), the workspace tools should return a clear error: "Save the document first to use workspace features."
**Warning signs:** Workspace MCP tools work for file-based use but fail for live canvas.

## Open Questions

1. **How should the Flow tab refresh when MCP tools modify flow files?**
   - What we know: MCP tools write directly to disk. The Flow tab needs to re-read files.
   - What's unclear: Push notification vs polling vs manual refresh button.
   - Recommendation: Use the existing MCP sync SSE pattern (`mcp-sync-state.ts`) to broadcast a `workspace-changed` event when any workspace MCP tool writes a file. The Flow tab subscribes and re-fetches. Fallback: a simple "Refresh" button.

2. **Dark mode theming for mermaid diagrams**
   - What we know: Mermaid supports `theme: 'dark'`. The WorkflowPanel uses `theme: 'default'`.
   - What's unclear: Whether to auto-switch mermaid theme based on app dark/light mode.
   - Recommendation: Initialize mermaid with dynamic theme matching `document.documentElement.classList.contains('light')` check. Can be deferred to polish.

3. **Maximum flow file size for browser rendering**
   - What we know: Mermaid can handle hundreds of nodes but becomes slow beyond ~500 nodes per diagram.
   - What's unclear: Real-world flow file sizes in PenBoard projects.
   - Recommendation: No limit initially. If performance issues arise, add a node count warning.
