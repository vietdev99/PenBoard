# Phase 07.2: Canvas Offthread Sync and Worker-Based Document Processing - Context

**Gathered:** 2026-03-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Prevent canvas heavy operations (syncFromDocument, render loop iteration) from blocking UI panels and interactions. When canvas loads or processes a large document (~70K nodes, ~500 frames), other UI panels (layers, properties, navigate) must remain responsive.

</domain>

<decisions>
## Implementation Decisions

### Sync Offloading — Chunked Yield (Worker deferred)
- **Approach**: Refactor `flattenToRenderNodes` into an async generator that yields every ~500 nodes, returning control to main thread between chunks
- **Why not Worker first**: `postMessage` serialization overhead (~200ms for 70K objects), `premeasureTextHeights` needs Canvas 2D context (unavailable in Worker without OffscreenCanvas), high re-architecture risk
- **Interruption handling**: If document changes mid-sync, abort current chunk sequence and restart with new data
- **Future**: If chunked yield insufficient, Web Worker is the Phase 07.3 escalation path
- **Scope**: `resolveRefs`, `resolveVarsDeep`, `flattenToRenderNodes`, `spatialIndex.rebuild` — all candidates for chunking

### Render Loop — Pre-filtered Visible List via Spatial Index
- **Approach**: When viewport changes, query existing R-tree spatial index for nodes in viewport → maintain `visibleRenderNodes` array → render loop iterates O(visible) instead of O(70K)
- **Why not frame budget**: Progressive render creates flickering — poor UX
- **Rebuild trigger**: `visibleRenderNodes` rebuilt when viewport (pan/zoom) changes significantly (debounced ~50ms)
- **Integration**: `spatialIndex.query(vpLeft, vpTop, vpRight, vpBottom)` already available

### Panel Isolation — Selective Subscriptions + Priority Scheduling
- **Selective subs**: Each panel component uses Zustand selectors to subscribe only to its relevant data slice (e.g., layers panel only watches `document.pages[activePageId].children`, property panel only watches `selection.selectedIds`)
- **Priority scheduling**: Non-critical panel updates use `requestIdleCallback` to defer behind canvas operations
- **Combined approach**: Selective subs fix root cause (unnecessary re-renders), priority scheduling adds safety net for remaining heavy updates

### Frame Thumbnail LOD — Cached Bitmap per Root Frame at Low Zoom
- **Approach**: At zoom <30%, draw a pre-cached thumbnail bitmap of each root frame instead of rendering all children nodes. Reduces 70K drawNode calls to ~500 drawImage calls.
- **Capture**: When a frame is fully rendered at higher zoom (≥30%), capture its content as a CanvasKit SkImage via clipped `surface.makeImageSnapshot()`
- **Trigger**: Uses both zoom level (<30%) AND visible node count as criteria — if viewport contains many visible nodes, thumbnails kick in even at slightly higher zoom
- **Quality**: Blurry acceptable at low zoom (Google Maps style). No one edits at <30% zoom — this is overview/navigation mode.
- **Invalidation**: When any node inside a frame changes → invalidate that frame's thumbnail, re-capture on next high-zoom render
- **Memory**: ~500 thumbnails × ~200KB ≈ 100MB WASM memory. Must call `SkImage.delete()` on invalidation to prevent leaks.
- **User insight**: "ở chế độ <30% là có thể thumbnail được rồi" — user confirmed threshold

### Claude's Discretion
- Exact chunk size for yield (500 nodes suggested, tune based on profiling)
- Spatial index query debounce timing (50ms suggested)
- Which panels to audit first (layers and property panels likely highest impact)
- Whether to use `requestIdleCallback` or `scheduler.postTask()` for priority scheduling
- Thumbnail resolution (full DPR or half-resolution for memory savings)
- Exact visible-node-count threshold for thumbnail trigger

</decisions>

## Specific Ideas

- User observed: "canvas mà load gì đó thì các panel thao tác khác cũng đơ luôn" — panels freeze alongside canvas
- User observed: "zoom xuống dưới <10%, hiện tượng giật lag lại bắt đầu" — 500 frames at low zoom still causes jank
- User wants independent panel operability — should be able to click layers, type properties even during canvas sync
- User insight: "ở chế độ zoom cỡ đó, chẳng ai sửa được cả, sao chúng ta không hiện trên frame kiểu 1 cái ảnh đã gen từ trước" — thumbnail for frames at low zoom
- User confirmed: "<30% là có thể thumbnail được rồi" — threshold for thumbnail mode

</specifics>
