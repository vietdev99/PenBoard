# Phase 7: Interactive Preview - Context

**Gathered:** 2026-03-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Users can preview their designs as interactive HTML prototypes in a separate browser tab. Preview supports navigation between screens via ScreenConnections, displays live sample data from bound entities, and runs in a sandboxed environment with strict CSP. Hot reload keeps preview synced with editor changes in real-time.

Requirements: PREV-01, PREV-02, PREV-03, PREV-04

</domain>

<decisions>
## Implementation Decisions

### Preview Trigger & Entry Point
- **Top bar button** (cáşˇnh Play icon) â€” always visible, one-click to preview
- **Browser tab má»›i** via window.open(). In Electron: shell.openExternal() to default browser
- **Preview scope**: selected frame if available, otherwise active page. Only `screen` pages are previewable (ERD/component pages not previewable)
- **Multiple preview tabs** allowed simultaneously
- **No live-update on page switch** â€” preview keeps its current page when editor switches pages. User navigates within preview independently

### Hot Reload
- **BroadcastChannel API** for communication between editor and preview tab
- **Debounce 500-1000ms** after each change before pushing update
- Since BroadcastChannel requires same origin, preview is served via **Nitro server route** (not blob URL)
- Editor POSTs preview data to Nitro, stores **in-memory** (no disk I/O)
- **SSE** from Nitro signals preview tab to re-render
- **Replace in-memory data** each reload â€” no accumulation, only current session data

### Navigation Behavior
- **Transition effects**: push â†’ slide left, modal â†’ fade overlay with backdrop dim (click backdrop/X to close), replace â†’ instant swap
- **Hover state**: cursor pointer + subtle highlight border/shadow on clickable elements
- **Back navigation**: browser back + toolbar back button. Push adds to history stack, replace replaces current
- **Trigger fidelity**: giáşŁ láş­p Ä‘Ăşng trigger â€” click â†’ click, hover â†’ fire on hover, submit â†’ fire on Enter/form submit
- **Hash routing** `#page-id` â€” URL changes on navigation, browser back works natively
- **Only explicit ScreenConnections** create navigation â€” no implicit linking from naming
- **targetFrameId**: scroll to that frame within target page after navigation
- **Missing page**: placeholder "Page not found" + Back button
- **Circular connections**: allowed â€” user navigates loops naturally, browser back handles unwinding
- **Hotspot mode**: toggle button in toolbar to highlight all clickable elements + show connection arrows

### Data Display
- **Tables**: full HTML `<table>` with headers from field names and rows from entity sample data, styled per design
- **Dropdowns**: clickable, opens with options from entity rows. Select-able but state not persisted
- **Text nodes with binding**: show sample data from first entity row (consistent with canvas)
- **Empty entity (no sample rows)**: keep original placeholder content (Phase 5 decision carried forward)
- **Images**: inline base64 or blob URL â€” self-contained, no external fetches
- **Icons (Lucide)**: inline SVG preserving icon and color
- **Components (RefNode)**: fully resolved HTML with arguments applied (same as canvas rendering)
- **Variables**: CSS custom properties + `:root` block with concrete values. Follow document's default theme variant
- **Fonts**: follow theme/font settings from PenBoard document. If custom fonts used, include them; system fonts stay as system fonts

### Sandbox & Security (PREV-04)
- **Nitro server route** (e.g., `/preview/:id`) instead of blob URL â€” enables BroadcastChannel same-origin communication
- **Strict CSP via `<meta>` tag** â€” block external scripts/fonts/images, only allow inline styles and PenBoard-generated inline scripts
- **JS allowed**: only JavaScript generated by PenBoard (navigation, transitions, dropdown interactions, form interactions). No user-defined scripts
- **No authentication** â€” local-only app, preview route serves localhost only
- **Self-contained HTML** â€” all resources inline (CSS, images as base64, SVGs inline). External resources blocked by CSP

### Responsive & Sizing
- **Viewport follows frame size** of the design (e.g., 375x812 for mobile, 1440x900 for desktop)
- **No viewport selector** â€” preview renders at design size. Want mobile? Design a mobile frame
- **Overflow**: browser scroll if content exceeds viewport
- **Lazy render per page** for large documents â€” only generate HTML for current page, fetch target page HTML on navigation

### Error Handling
- **Graceful fallback**: missing font â†’ system font, broken image â†’ placeholder, invalid CSS â†’ skip
- **Circular connections**: allowed, no detection needed
- **Large documents**: lazy page rendering avoids loading all pages at once

### Accessibility
- **Semantic HTML from node roles**: button â†’ `<button>`, nav â†’ `<nav>`, heading â†’ `<h1-h6>`, no role â†’ `<div>`
- **Keyboard navigation**: Tab focus on clickable elements (tabindex), Enter/Space fires navigation
- **Form elements**: text inputs type-able, checkboxes/radios toggle-able. No state persistence, no form submission

### Preview Toolbar
- **Layout**: top bar â€” Back (left) | Breadcrumb (center) | Hotspots toggle + Refresh (right)
- **Breadcrumb**: "Page name > Frame name" â€” clickable to navigate
- **Toggle visibility**: toggle button to hide/show toolbar
- **Toolbar theme**: follows system OS dark/light mode (not document theme)

### Theme in Preview
- **Default theme variant** from document (CSS variables from default variant of each theme axis)
- No theme switcher in preview toolbar (v1.1 scope)

### Export
- **Download HTML button** in toolbar â€” exports self-contained .html file with all pages, navigation JS, CSS
- **Local-only** â€” no share links. Want to share? Export HTML file and send

### Form Interactions
- **Basic interactive**: text inputs typeable, checkboxes checkable, radio buttons toggleable
- No state persistence, no form validation, no submission handling

### Loading UX
- **Skeleton + progress bar** when generating preview. Tab opens immediately with skeleton, content renders when ready

### Claude's Discretion
- Exact CSS transition durations and easing for navigation animations
- Exact debounce timing within 500-1000ms range
- Skeleton loading design
- BroadcastChannel message format
- Nitro route naming and API shape
- SSE event naming
- Exact toolbar styling and icon choices
- Progress bar implementation
- HTML export bundling approach (single file vs zip)

</decisions>

## Specific Ideas

- Preview toolbar should feel like Figma's prototype toolbar â€” compact, dark, minimal
- Hotspot mode should clearly highlight all interactive elements with a distinct overlay color (like Figma's blue hotspot highlights)
- Navigation transitions should be smooth but not slow â€” fast animations (200-300ms) that communicate the transition type without feeling sluggish
- The Download HTML feature should produce a truly self-contained file that works when opened from desktop without any server
