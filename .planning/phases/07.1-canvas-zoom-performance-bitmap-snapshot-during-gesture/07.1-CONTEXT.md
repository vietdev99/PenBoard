# Phase 07.1: Canvas Zoom Performance — Bitmap Snapshot During Gesture - Context

**Gathered:** 2026-03-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Eliminate canvas freeze during zoom/pan on large documents (70K+ nodes). Replace per-frame re-rendering with a bitmap snapshot approach: capture rendered frame as bitmap, display transformed bitmap during zoom/pan gestures, re-render at full fidelity when gesture ends. Also finalize the loading modal progress feature and clean up prior quick-task fixes.

</domain>

<decisions>
## Implementation Decisions

### Render Model — Hybrid
- **Document small** (render time <16ms): Live render as current behavior. No bitmap overhead.
- **Document large** (render time >16ms): Bitmap-first mode. Canvas displays bitmap; only re-renders when document changes or gesture ends.
- **Auto-detect threshold**: Benchmark first render after loadDocument/syncFromDocument. If >16ms → enable bitmap mode.
- **Re-evaluate on page change**: Each page may have different node counts. Re-benchmark when activePageId changes.

### Snapshot Source & Lifecycle
- **Source**: CanvasKit `surface.makeImageSnapshot()` — reads directly from GPU buffer, pixel-perfect.
- **Resolution**: Canvas size × devicePixelRatio (e.g., 1920×1080 × 2 = 3840×2160 pixels).
- **Invalidation**: Bitmap invalidated when document changes (edit, AI generate, undo/redo). After next full render completes → capture new snapshot.
- **No stale display**: When standing still, always live render. Bitmap only used during active gesture.

### Gesture Scope
- **Zoom/pan only**: Bitmap used during wheel zoom and space+drag pan gestures.
- **Drag/resize/rotate**: Always live render — user is interacting directly with nodes, bitmap would show wrong positions.
- **Click**: Ends gesture immediately, triggers full re-render. User needs accurate selection handles.

### Edit Behavior in Bitmap Mode
- **During edit**: Show bitmap background + live selection overlay for the node being edited.
- **Debounced full re-render**: After 200ms idle from last edit, full re-render + capture new snapshot.
- **Continuous edits** (dragging color slider, resizing): Selection overlay updates live, background bitmap stays until 200ms after user stops.

### Gesture End Behavior
- **Idle threshold**: 150ms without zoom/pan input → gesture considered ended.
- **Re-render**: Full fidelity render at new zoom/pan level.
- **Swap**: Instant swap from bitmap to live render (no fade/crossfade animation).

### Zoom Quality
- **Zoom in**: Accept temporary blur (bitmap scaled up). Lasts only until re-render completes (~150ms after gesture ends). Google Maps-style behavior.
- **Zoom out**: No quality issue (bitmap scaled down = good quality).
- **No multi-resolution**: Single snapshot at current resolution. Simpler, less memory.

### Overlays During Gesture
- **All overlays freeze into bitmap**: Selection handles, frame labels, connection arrows, component badges — all captured in snapshot and transform with it.
- **After gesture**: Re-render draws everything at correct new positions/sizes.
- **Guides and marquee**: Not relevant during zoom/pan (only appear during drag).

### Scope & Cleanup
- **Keep**: WASM leak fixes from quick task (8+ `.delete()` calls in skia-renderer.ts, skia-overlays.ts, skia-erd-renderer.ts). These prevent memory leaks regardless of bitmap approach.
- **Rewrite**: Wheel event throttle and `isPanning` logic — will be replaced by bitmap approach's gesture detection.
- **Remove**: All diagnostic `console.warn` logs (frame time, syncFromDocument time).
- **Finalize**: Loading modal progress (Opening → Processing → Drawing) — fix timing bug where `onNextRender` fires before `syncFromDocument`.
- **willReadFrequently**: Already fixed in skia-overlays.ts.

### Claude's Discretion
- Exact bitmap caching data structure (Map, WeakRef, single field)
- How to draw bitmap during gesture (CanvasKit `drawImage` with viewport matrix, or CSS transform on HTML element)
- syncFromDocument change detection optimization details
- Exact benchmark methodology (single render timing vs average)

</decisions>

<specifics>
## Specific Ideas

- User wants Pencil-like behavior: components show update animation (blue sweep) when being modified, bitmap updates after completion
- "chụp ảnh svg đó để ở chế độ zoom nào đó, load ảnh thôi chứ không cần zoom in out của canvas" — static image during zoom, not live re-rendering
- "thường thì ở chế độ fit, hoặc 2-3 view mới là bước sửa" — fit/overview is the problematic zoom level (all nodes visible)
- "zoom nhỏ quá thì chẳng cần sửa" — when zoomed in close, few nodes visible, no performance issue

</specifics>

<deferred>
## Deferred Ideas

- **Partial re-render (dirty rectangle)** — Only re-render changed region and composite onto bitmap. Discussed but deferred due to CanvasKit immediate-mode complexity (z-order, shadows, clipping). Could be Phase 07.2 if debounced full re-render proves too slow for specific workflows.
- **Multi-resolution snapshots** — Capture at 2-3 zoom levels for less blur during zoom. Deferred as single snapshot with temporary blur is acceptable.
- **Viewport-based hybrid render** — Live render nodes in viewport, bitmap for out-of-viewport. Simpler than partial render but still adds complexity. Not needed if bitmap approach solves the core issue.

</deferred>
