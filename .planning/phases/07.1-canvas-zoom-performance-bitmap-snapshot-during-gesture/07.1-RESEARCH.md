# Phase 07.1: Canvas Zoom Performance -- Bitmap Snapshot During Gesture - Research

**Researched:** 2026-03-19
**Domain:** CanvasKit/Skia WASM bitmap caching, GPU rendering performance, gesture-based rendering optimization
**Confidence:** HIGH

## Summary

This phase addresses a critical performance bottleneck: canvas freezing during zoom/pan on large documents (70K+ nodes). The solution is a bitmap snapshot approach where the rendered frame is captured via CanvasKit's `surface.makeImageSnapshot()`, displayed as a transformed bitmap during active zoom/pan gestures, and replaced with a full-fidelity render when the gesture ends.

The existing codebase already has foundational elements: `isPanning` flag with 150ms idle timeout, RAF-throttled wheel events, viewport culling, text LOD, and pre-built render node lists. The bitmap snapshot approach builds on top of these patterns, replacing the per-frame full render during gestures with a single `drawImage()` call that is essentially free.

**Primary recommendation:** Add bitmap snapshot capture/display to `SkiaEngine.render()` method. After each full render + `surface.flush()`, capture `surface.makeImageSnapshot()`. During active zoom/pan gestures, skip the full render path and instead draw the cached bitmap with the new viewport transform applied. Re-render at full fidelity once the gesture idle timer (150ms) fires.

## Architecture Patterns

### Bitmap Snapshot State Machine

The render method operates in one of three modes:

```
LIVE_RENDER (default)
  |
  |-- render time > 16ms detected --> BITMAP_ENABLED
  |                                      |
  |                                      |-- gesture starts (zoom/pan) --> BITMAP_GESTURE
  |                                      |                                    |
  |                                      |                                    |-- gesture idle 150ms --> BITMAP_ENABLED
  |                                      |                                    |    (full re-render + capture new snapshot)
  |                                      |
  |                                      |-- document changes --> invalidate snapshot
  |                                      |    (next full render captures new one)
  |                                      |
  |                                      |-- page change --> re-benchmark
  |                                           (may switch back to LIVE_RENDER)
```

### Pattern 1: Bitmap Snapshot Capture After Full Render
**What:** After `surface.flush()` in a full render, call `surface.makeImageSnapshot()` and store the result. Also store the viewport state (zoom, panX, panY) at capture time.
**When to use:** Every full render when bitmap mode is enabled.
**Example:**
```typescript
// In skia-engine.ts render() method, after surface.flush():
// Source: canvaskit-wasm ^0.40.0 types/index.d.ts line 3049
canvas.restore()
this.surface.flush()

// Capture snapshot for bitmap mode
if (this.bitmapEnabled) {
  // Delete previous snapshot to prevent WASM memory leak
  this.cachedSnapshot?.delete()
  this.cachedSnapshot = this.surface.makeImageSnapshot()
  this.snapshotViewport = { zoom: this.zoom, panX: this.panX, panY: this.panY }
}
```

### Pattern 2: Bitmap Display During Gesture
**What:** When gesturing and bitmap is available, skip the full node render loop. Instead, clear canvas, apply new viewport transform relative to the snapshot's original viewport, and draw the bitmap image.
**When to use:** When `isPanning === true && cachedSnapshot !== null && bitmapEnabled`.
**Example:**
```typescript
// In skia-engine.ts render() method, early return path:
private render() {
  if (!this.surface || !this.canvasEl) return

  // Bitmap fast path: during active gesture, draw cached snapshot
  if (this.bitmapEnabled && this.isGesturing && this.cachedSnapshot) {
    this.renderBitmapFastPath()
    return
  }
  // ... full render path continues
}

private renderBitmapFastPath() {
  const canvas = this.surface!.getCanvas()
  const ck = this.ck
  const dpr = window.devicePixelRatio || 1

  canvas.clear(parseColor(ck, getCanvasBackground()))
  canvas.save()
  canvas.scale(dpr, dpr)

  // Compute relative transform: new viewport relative to snapshot viewport
  // The snapshot was captured at snapshotViewport (sZoom, sPanX, sPanY)
  // We want to display it at current viewport (zoom, panX, panY)
  const sv = this.snapshotViewport!
  const scaleRatio = this.zoom / sv.zoom
  const offsetX = this.panX - sv.panX * scaleRatio
  const offsetY = this.panY - sv.panY * scaleRatio

  const imgW = this.cachedSnapshot!.width() / dpr
  const imgH = this.cachedSnapshot!.height() / dpr

  const paint = new ck.Paint()
  canvas.drawImageRect(
    this.cachedSnapshot!,
    ck.XYWHRect(0, 0, this.cachedSnapshot!.width(), this.cachedSnapshot!.height()),
    ck.XYWHRect(offsetX, offsetY, imgW * scaleRatio, imgH * scaleRatio),
    paint,
  )
  paint.delete()

  canvas.restore()
  this.surface!.flush()
}
```

### Pattern 3: Auto-Detection Benchmark
**What:** After `syncFromDocument()` or page change, time the next full render. If it exceeds 16ms, enable bitmap mode. If under 16ms, disable it.
**When to use:** After document load, page switch, or significant document changes.
**Example:**
```typescript
// In render(), after a full render cycle:
const elapsed = performance.now() - _t0
if (this.benchmarkPending) {
  this.bitmapEnabled = elapsed > 16
  this.benchmarkPending = false
}
```

### Anti-Patterns to Avoid
- **Never keep stale snapshots while standing still:** When not gesturing, always do full live render. The bitmap is only for gesture smoothness.
- **Never forget to .delete() old snapshots:** CanvasKit Images are WASM objects. Failing to delete causes memory leaks. Always delete the old snapshot before capturing a new one.
- **Never apply bitmap mode during drag/resize/rotate:** User is interacting with specific nodes. Bitmap would show wrong positions. Only zoom/pan use bitmap.
- **Never skip the full render after gesture ends:** The 150ms idle timer MUST trigger a full re-render to show accurate positions at the new zoom level.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| GPU buffer capture | Manual pixel readback via getImageData | `surface.makeImageSnapshot()` | GPU-to-GPU copy is instant; CPU readback would be 100x slower |
| Viewport transform math | Custom matrix multiplication | Existing `viewportMatrix()` in skia-viewport.ts | Already tested, handles zoom+pan correctly |
| Gesture idle detection | Custom mouse event tracking | Extend existing `isPanning` + `panIdleTimer` pattern | Already proven in codebase with 150ms threshold |
| Image drawing | Manual pixel manipulation | CanvasKit `canvas.drawImageRect()` | GPU-accelerated image scaling, handles any viewport transform |

**Key insight:** CanvasKit's `makeImageSnapshot()` returns a GPU-backed Image that can be drawn via `drawImageRect()` with near-zero cost. This is the fundamental enabler -- the capture and display are both GPU operations that bypass the CPU entirely.

## Common Pitfalls

### Pitfall 1: WASM Memory Leak from SkImage
**What goes wrong:** Each `makeImageSnapshot()` allocates a WASM Image object. Without explicit `.delete()`, these accumulate and eventually crash the tab.
**Why it happens:** JavaScript GC doesn't manage WASM heap. CanvasKit objects need manual disposal.
**How to avoid:** Always call `this.cachedSnapshot?.delete()` before reassigning. Add cleanup in `dispose()`.
**Warning signs:** Browser tab memory steadily increasing during zoom/pan interactions.

### Pitfall 2: Bitmap Transform Math Off by DPR
**What goes wrong:** The bitmap appears offset or at wrong scale because devicePixelRatio wasn't accounted for.
**Why it happens:** `makeImageSnapshot()` captures at physical pixel resolution (canvas.width x canvas.height), but the viewport transform operates in CSS pixel space.
**How to avoid:** The snapshot image dimensions are `canvasEl.width x canvasEl.height` (physical pixels). When drawing back, divide by DPR to get CSS pixel dimensions, then apply the relative viewport transform.
**Warning signs:** Bitmap appears 2x too large or 2x offset on high-DPR displays.

### Pitfall 3: Gesture Detection Conflict with Existing isPanning
**What goes wrong:** The bitmap mode and existing `isPanning` flag get out of sync, causing either: (a) bitmap displayed when it shouldn't be, or (b) full render during gesture.
**Why it happens:** Currently `isPanning` is set in `setViewport()` and cleared by 150ms timer. Bitmap mode needs to know about gestures too.
**How to avoid:** Unify gesture state. The `setViewport()` method already sets `isPanning = true` and starts the idle timer. Bitmap render path should check `isPanning` directly. The existing 150ms timer that sets `isPanning = false` naturally triggers the full re-render.
**Warning signs:** Stuttering during zoom/pan, or bitmap showing during drag operations.

### Pitfall 4: loadDocumentWithProgress Timing Bug
**What goes wrong:** Loading modal shows "Drawing graphics..." but `onNextRender` fires before `syncFromDocument()` completes, dismissing the modal prematurely.
**Why it happens:** In `skia-engine-ref.ts` line 68-73, `requestAnimationFrame` is used to wait for Zustand subscription to trigger `syncFromDocument()`, but this races with the RAF.
**How to avoid:** The current approach uses 1 RAF to wait for `syncFromDocument`. The fix is to either: (a) add an explicit callback after `syncFromDocument()` completes, or (b) use `onNextRender` which is fired after `render()` completes (which runs after syncFromDocument). The issue is that `markDirty()` in syncFromDocument doesn't guarantee the very next RAF will do a full render if the dirty flag was already consumed.
**Warning signs:** Modal flashes briefly and disappears before canvas is actually rendered.

### Pitfall 5: Snapshot Invalidation Race During AI Generation
**What goes wrong:** AI generation inserts nodes rapidly. Each insertion triggers syncFromDocument + markDirty. If user zooms during generation, the bitmap might be from a stale document state.
**Why it happens:** Bitmap invalidation triggers on document change, but re-capture only happens after a full render. If the user is gesturing, no full render occurs.
**How to avoid:** Set a `snapshotStale` flag when document changes during active gesture. When gesture ends and full re-render happens, the new snapshot is automatically captured. The stale bitmap is acceptable during gesture (user is zooming, not looking at details).
**Warning signs:** After zoom/pan ends during AI generation, there's a brief flash of old content before re-render completes.

## Open Questions

1. **CanvasKit drawImage vs CSS Transform for bitmap display**
   - What we know: CanvasKit `drawImage`/`drawImageRect` works within the same WebGL pipeline and is very fast. CSS transform on a separate DOM element would require extracting the bitmap to an HTML Image and positioning it above the canvas.
   - What's unclear: Whether CSS transform could provide smoother sub-frame interpolation
   - Recommendation: Use CanvasKit `drawImageRect` -- keeps everything in one pipeline, avoids DOM manipulation complexity, and the existing render loop already handles the timing. **HIGH confidence** this is the right approach.

2. **Benchmark methodology for auto-detection**
   - What we know: A single render timing after loadDocument may not be representative (cold caches, async image loading).
   - What's unclear: Whether to use single measurement or rolling average.
   - Recommendation: Use the first render after syncFromDocument as the benchmark (warm enough -- fonts loaded, text measured). If render time > 16ms, enable bitmap mode. Re-benchmark on page change. Simple and deterministic. Can be refined later if needed.

3. **Snapshot invalidation granularity during rapid edits**
   - What we know: The 200ms debounce after edit means the bitmap background stays stale for up to 200ms while user drags a color slider.
   - What's unclear: Whether 200ms is perceptible (it's less than the document changes during a color slider drag).
   - Recommendation: 200ms debounce is fine per user decision. The selection overlay renders live over the bitmap background, so the user sees immediate feedback on the selected element.
