```typescript
/**
 * Bitmap fast path: draw cached snapshot with viewport transform applied.
 * Used during active zoom/pan gesture when bitmapEnabled is true.
 * Near-zero cost тАФ single GPU drawImageRect call instead of full node iteration.
 */
private renderBitmapFastPath() {
  const canvas = this.surface!.getCanvas()
  const ck = this.ck
  const dpr = window.devicePixelRatio || 1

  canvas.clear(parseColor(ck, getCanvasBackground()))
  canvas.save()
  canvas.scale(dpr, dpr)

  // Compute relative transform: new viewport relative to snapshot viewport
  const sv = this.snapshotViewport!
  const scaleRatio = this.zoom / sv.zoom
  const offsetX = this.panX - sv.panX * scaleRatio
  const offsetY = this.panY - sv.panY * scaleRatio

  const imgW = this.cachedSnapshot!.width() / dpr
  const imgH = this.cachedSnapshot!.height() / dpr

  const paint = new ck.Paint()
  canvas.drawImageRect(
    this.cachedSnapshot!,
    ck.XYWHRect(0, 0, this.cachedSnapshot!.width(), this.cachedSnapshot!.height()),
    ck.XYWHRect(offsetX, offsetY, imgW * scaleRatio, imgH * scaleRatio),
    paint,
  )
  paint.delete()

  canvas.restore()
  this.surface!.flush()
}
```

**1. Fix `loadDocumentWithProgress` timing bug in `src/canvas/skia-engine-ref.ts`:**

The bug: `requestAnimationFrame` at line 69 waits 1 frame for Zustand subscription to trigger `syncFromDocument()`, but this races тАФ the dirty flag from the prior `markDirty()` may already be consumed in the same RAF, causing `onNextRender` to fire before `syncFromDocument` completes its next render cycle.

Replace the inner `setTimeout` callback body (lines 61-86) with:

```typescript
setTimeout(() => {
  try {
    setFileLoading({ open: true, name: fileName, status: 'Processing document...' })
    loadFn()
    setFileLoading({ open: true, name: fileName, status: 'Drawing graphics...' })

    // Wait for syncFromDocument to process (triggered by Zustand subscription)
    // then wait for the subsequent render to complete before dismissing modal.
    // Use 2 RAFs: first waits for syncFromDocument, second waits for render.
    requestAnimationFrame(() => {
      requestAnimationFrame(() => {
        if (_engine) {
          _engine.onNextRender(() => {
            zoomToFitContent()
            requestAnimationFrame(() => useCanvasStore.getState().setFileLoading(null))
          })
          // Force a fresh render cycle in case dirty flag was already consumed
          _engine.markDirty()
        } else {
          requestAnimationFrame(() => {
            zoomToFitContent()
            useCanvasStore.getState().setFileLoading(null)
          })
        }
      })
    })
  } catch (err) {
    console.error('[loadDocumentWithProgress] Error:', err)
    useCanvasStore.getState().setFileLoading(null)
  }
}, 50)
```

The key change: add a second `requestAnimationFrame` to ensure `syncFromDocument()` (triggered by Zustand subscription after `loadFn()`) has completed before registering `onNextRender`. Also call `_engine.markDirty()` explicitly to ensure a fresh render cycle fires.

**2. Simplify wheel handler in `src/canvas/skia/skia-canvas.tsx` (lines ~370-430):**

The existing RAF-throttled wheel handler already works correctly and efficiently. The CONTEXT.md says to "rewrite wheel event throttle and isPanning logic -- will be replaced by bitmap approach's gesture detection." However, the wheel handler doesn't manage `isPanning` тАФ that's done in `setViewport()` in `skia-engine.ts`. The wheel handler merely accumulates deltas and calls `engine.zoomToPoint()` / `engine.pan()`.

The simplification is: remove the `pendingZoom`/`pendingPan` accumulation pattern. With bitmap mode, per-frame zoom/pan calls are cheap (just draw a bitmap), so the RAF coalescing is unnecessary. Replace with direct calls:

Replace the wheel useEffect (lines ~373-430) with:

```typescript
// Wheel: zoom + pan
useEffect(() => {
  const canvasEl = canvasRef.current
  if (!canvasEl) return

  const handleWheel = (e: WheelEvent) => {
    e.preventDefault()
    e.stopPropagation()
    const engine = engineRef.current
    if (!engine) return

    if (e.ctrlKey || e.metaKey) {
      // Pinch-to-zoom or Ctrl+scroll
      let delta = -e.deltaY
      if (e.deltaMode === 1) delta *= 40
      const factor = Math.pow(1.005, delta)
      const newZoom = engine.zoom * factor
      engine.zoomToPoint(e.clientX, e.clientY, newZoom)
    } else {
      // Two-finger pan or regular scroll
      let dx = -e.deltaX
      let dy = -e.deltaY
      if (e.deltaMode === 1) { dx *= 40; dy *= 40 }
      engine.pan(dx, dy)
    }
  }

  canvasEl.addEventListener('wheel', handleWheel, { passive: false })
  return () => {
    canvasEl.removeEventListener('wheel', handleWheel)
  }
}, [])
```

This removes the `pendingZoom`, `pendingPanDx`, `pendingPanDy`, `wheelRaf`, and `flushWheel` variables. Each wheel event directly calls the engine, which sets `isPanning = true` and triggers the bitmap fast path via `markDirty()`. The engine's own render loop (RAF-based dirty flag) naturally coalesces multiple `markDirty()` calls within a single frame.

NOTE: If testing reveals that high-frequency trackpad events cause too many Zustand `setViewportBatch` calls, the RAF coalescing can be restored. But with bitmap mode the render cost per frame is negligible, so the Zustand update is the only concern, and it's already batched via `setViewportBatch`.
